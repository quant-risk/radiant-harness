# Radiant Harness вЂ” Complete Harness Plan

**Date**: 2026-06-24
**Status**: Approved direction (post-discovery)
**Target**: Merge the methodology of [spec-driven](https://github.com/igoruehara/spec-driven) (community-validated Lean/DDD/TDD/SDD workflow) with radiant-harness's cross-platform, multi-LLM execution engine вЂ” producing a **vendor-neutral, agent-agnostic, complete SDD harness**.

---

## 1. Why we're doing this

**Current state (v0.3.5)**:
- Strong on execution: code generation, gate running, multi-LLM routing, cross-platform, 188 tests, vendor-neutrality, security hardening
- Weak on methodology: assumes spec.md + tasks.md exist; doesn't help with discovery, design, domain modeling, session continuity, audit, metrics, PR review

**spec-driven (community, 137в­ђ)**:
- Strong on methodology: 15 skills covering Lean Inception в†’ DDD в†’ TDD/RFC в†’ SDD, tier system, ADRs, state machine, audit, metrics, multi-agent view generation
- Weak on execution: stops at the spec; doesn't generate code

**The merge**: a single CLI that gives you both вЂ” the full methodology PLUS the engine that turns specs into code, validated code, and merged PRs. Best of both worlds.

---

## 2. The architectural principle (NON-NEGOTIABLE)

> **The format of skills and artifacts must be so good that any agent on the market can consume them without plugin, hook, or vendor namespace.**

**Test of fire**: if tomorrow we delete every native view (`.claude/`, `.cursor/`, `.windsurf/`, etc.) and only `AGENTS.md` + `.radiant-harness/skills/` remain, **any modern LLM IDE must be able to work on the project**.

**Consequences**:
- вќЊ NO `CLAUDE.md` as a special-cased file name
- вќЊ NO `.claude/skills/` namespace as the primary location
- вќЊ NO Claude-Code-specific hooks as required infrastructure
- вќЊ NO slash-command syntax (`/kickoff`, `/nova-feature`) as the only entry point вЂ” slash commands are Claude-Code convention; CLI commands are universal
- вњ… Skills live in `.radiant-harness/skills/<name>/SKILL.md` вЂ” plain Markdown + YAML frontmatter, parseable by any LLM
- вњ… `AGENTS.md` at project root is the canonical universal index (already recognized by Codex, Cursor, Copilot, Gemini, Windsurf)
- вњ… Native views (`.claude/skills/`, `.cursor/rules/`, etc.) are **OPT-IN** and only generated when `--agent=<list>` is specified
- вњ… Each skill includes a YAML frontmatter declaring its contract (inputs, outputs, gates) вЂ” machine-readable by any LLM, not just Claude

### 3.1 What gets generated by `radiant init`

```bash
# Default: only universal artifacts
radiant init meu-app
# в†’ AGENTS.md
# в†’ .radiant-harness/skills/<15 skills>/
# в†’ specs/_templates/
# в†’ docs/_templates/

# Opt-in: add native views for specific agents
radiant init meu-app --agent=claude,cursor,codex
# в†’ (everything above, plus)
# в†’ .claude/skills/<name>/SKILL.md           (Claude Code enhanced runtime)
# в†’ .claude/settings.json                   (Claude-specific hooks, OPTIONAL)
# в†’ .cursor/rules/sdd.mdc                   (Cursor-specific rule file)
# в†’ AGENTS.md is already there               (Codex uses it natively)
```

The native views are **thin wrappers** around `.radiant-harness/skills/*/SKILL.md`. We don't maintain two skill repositories вЂ” the canonical version lives in `.radiant-harness/skills/`, and the native view is regenerated from it via `radiant update`.

### 3.2 Who consumes what

| Agent | Primary read | Native view (if requested) | Skill execution |
|-------|-------------|----------------------------|-----------------|
| Mavis Code | `AGENTS.md` + `.radiant-harness/skills/*/SKILL.md` | none (uses universal) | Reads SKILL.md, follows instructions |

**The "no AGENTS.md" world still works**: an LLM pointed at the project root will discover `.radiant-harness/skills/*/SKILL.md` via directory listing and follow the YAML frontmatter contract.

## 4. Skill schema (open spec)

Every skill follows this structure. The schema is published openly вЂ” any agent author can implement parsing.

### 4.1 Directory layout


### 4.4 Why this schema is superior to vendor-specific formats

| Property | Claude `.claude/skills/` | Open `AGENTS.md` + `.radiant-harness/skills/` |
|----------|--------------------------|---------------------------------------------|
| Vendor lock-in | Claude Code only | None |
| Slash command as entry | Required | Optional |
| Machine-readable contract | Implicit (frontmatter varies) | Explicit YAML (inputs/outputs/gates/version) |
| Tier system | None | Built-in |
| Update mechanism | Manual | `radiant update` |
| Distributable | Per-repo copy | Single binary contains all skills; `init` extracts |
| Cross-agent portability | None | Full |

## 8. Risk register

| Risk | Mitigation |
|------|------------|
| Schema bikeshedding | v1 ships with one author; iterate after feedback from real users |
| Skills become stale as agents evolve | `version` field in frontmatter; `radiant update` pulls new versions |
| Native view generation becomes maintenance burden | Generate from `.radiant-harness/skills/`, not from separate templates |
