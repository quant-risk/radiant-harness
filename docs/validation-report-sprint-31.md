### Autodata command details

`radiant autodata <skill-name> --domain="<description>"`:
- Reads `RADIANT_OPENAI_API_KEY` or `RADIANT_ANTHROPIC_API_KEY` (or
  falls back to `OPENAI_API_KEY` / `ANTHROPIC_API_KEY`)
- Calls LLM with a meta-prompt + the schema contract
- Parses response on `===FRONTMATTER===` / `===SKILLMD===` markers
- Writes `frontmatter.yaml` + `SKILL.md` to `--output` path
  (default `internal/skill/skills/<name>/`)
- `--dry-run` prints to stdout instead of writing files
- **No LLM key?** Falls back to a stub template with clear TODOs,
  so the command always produces useful output

**Limitation**: requires API key for full LLM generation. The
stub mode is intentionally useful — a hand-fill template is better
than failure.

## Iteration discipline

Five issues caught + fixed at dev time:

1. **YAML colon-in-description** (all 3 new skills): auto-fixed
   via Python script.
2. **YAML list-item indentation drift** (all 3 new skills):
   3 occurrences missed by Python script; fixed manually.
3. **`95%` literal in `fmt.Sprintf` format string** (eval body):
   escapes as `95%%`.
4. **`% o` from `20% of`** (eval body): vet parses `20%` as a
   format verb. Fix: escape as `20%%`.
5. **`gofmt` drift on main.go** (autodata additions): fixed
   via `gofmt -w`.

All caught at dev time, not at user time.
